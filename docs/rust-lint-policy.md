# Rust lint policy

quickdiff uses layered checks for code generated by humans and agents:

1. `rustfmt` for deterministic formatting.
2. Compiler and Clippy lints in `Cargo.toml` for high-signal correctness/style issues.
3. `clippy.toml` for MSRV-aware behavior and stricter Clippy analysis in tests.
4. `cargo-deny` for dependency/advisory/license/source hygiene.
5. `slop-scan` for heuristic AI-code smells and repo-shape checks.
6. `AGENTS.md` for pre-generation guidance.

## Local checks

```bash
cargo fmt-check
cargo lint
cargo test-all
cargo deny-check
npx -y slop-scan scan . --lint
```

or:

```bash
scripts/lint.sh
```

## Policy

- Fix lint findings at the source; avoid broad `#[allow(...)]`.
- If an allow is required, scope it to the smallest item and explain why inline.
- Production code should avoid `unwrap`, `expect`, `panic!`, `todo!`, `unimplemented!`, and `unreachable!`.
- Public library APIs should expose typed errors; binaries may use `anyhow` at orchestration boundaries.
- Prefer borrowed inputs when ownership is not required.
- Avoid clone-driven ownership fixes; reshape borrowing/data flow first.
- Avoid byte indexing/slicing of UTF-8 text; use char-safe APIs.
- Preserve quickdiff architecture: core stays UI-agnostic, loading remains lazy, rendering remains viewport-scoped.

## Tightening later

Current lint levels are intentionally compatible with the existing crate and MSRV. Future tightening candidates:

- `missing_docs`/`unreachable_pub` across all public items.
- selected conversion/indexing lints once hot paths are audited.
- stricter `slop-scan` rules after existing false positives are triaged.
